# Engineering Constitution

이 문서는 앞으로 이 코드베이스를 수정할 때 반드시 지켜야 하는 설계 원칙이다.

목표는 다음 3가지다.

- 상태 불일치 방지
- 숨겨진 fallback로 인한 사이드이펙트 방지
- 문제를 덮지 않고 드러내는 방향으로 구현

## 1. Single Source Of Truth

- 동일한 비즈니스 상태는 반드시 하나의 원본 필드만 가진다.
- 다른 필드에 같은 의미의 데이터가 저장되어야 한다면, 그 필드는 원본이 아니라 파생 인덱스다.
- 원본과 파생을 같은 급의 진실처럼 읽으면 안 된다.
- 읽기 로직은 원본만 기준으로 삼는다.

예시:

- 회사 참여사업의 원본은 `companies.programs`
- `programs.companyIds`는 파생 인덱스

## 2. Fallback로 깨진 상태를 숨기지 않는다

- 데이터가 비정상일 때, fallback로 “그럴듯하게” 보이게 만들지 않는다.
- fallback가 single source of truth를 우회하면 금지한다.
- fallback는 오직 화면 로딩 중 임시 표시, 구버전 호환 마이그레이션의 짧은 전환기에서만 허용한다.
- fallback를 넣어야 할 때는 반드시 제거 계획과 종료 시점을 문서화한다.

금지 예시:

- 원본 필드가 비어 있으면 다른 컬렉션을 뒤져서 임의로 상태를 복원하는 방식
- 관리자 화면은 A 필드, 사용자 화면은 B 필드를 읽는 방식

허용 예시:

- 배포 직후 짧은 마이그레이션 기간 동안만 쓰는 임시 호환 로직
- 단, TODO가 아니라 실제 제거 계획과 데이터 정리 작업이 같이 있어야 한다

## 3. 에러는 숨기지 말고 드러낸다

- 데이터가 깨졌는데도 조용히 진행시키지 않는다.
- 비즈니스상 중요한 상태가 없거나 모순되면, 에러를 노출하거나 저장을 막는다.
- “조용히 빈 배열”, “조용히 기본값”으로 덮는 방식은 신중하게 제한한다.
- 문제가 감지되면 사용자용 메시지와 개발자용 로그를 남긴다.

원칙:

- 사용자에게는 이해 가능한 메시지
- 개발자에게는 원인 추적이 가능한 로그
- 데이터가 잘못된 상태로 다음 단계로 넘어가지 않게 차단

## 4. 쓰기 경로는 반드시 중앙화한다

- 같은 상태를 바꾸는 코드가 여러 군데 흩어지면 안 된다.
- 비즈니스 상태 변경은 공통 유틸, 서비스, 함수 한 군데를 통해서만 수행한다.
- 컴포넌트에서 직접 여러 문서를 제각각 수정하지 않는다.
- 관리자 화면, 회사 화면, 승인 플로우가 같은 상태를 바꾼다면 반드시 같은 쓰기 함수를 호출해야 한다.

예시:

- `replaceCompanyPrograms`
- `replaceProgramCompanies`

## 5. 파생 인덱스는 원본에서 항상 재생성 가능해야 한다

- 파생 데이터는 언제든 원본으로부터 다시 만들 수 있어야 한다.
- 파생 데이터가 깨져도 시스템의 진실이 바뀌어서는 안 된다.
- 파생 필드를 읽더라도, 권한 판단이나 신청 가능 여부 같은 핵심 비즈니스 판단에는 사용하지 않는다.

## 6. 읽기 기준은 화면마다 달라지면 안 된다

- 같은 비즈니스 개념을 화면마다 다른 기준으로 읽는 것을 금지한다.
- “관리자 화면은 A, 회사 화면은 B” 구조는 장기적으로 반드시 깨진다.
- 한 개념에 대해 읽기 기준이 바뀌면 관련 화면 전부를 함께 수정한다.

체크 질문:

- 이 값은 관리자와 회사가 같은 기준으로 보고 있는가
- 대시보드, 상세, 신청, 통계가 같은 원본을 읽는가

## 7. UI 상태와 저장 상태를 분리하되, 저장 성공 후 즉시 일치시킨다

- 편집 모달의 로컬 상태는 허용한다.
- 그러나 저장 성공 후 UI가 이전 상태를 계속 보여주면 안 된다.
- 로컬 상태를 쓰는 경우, 저장 성공 시 즉시 최신 상태로 반영한다.
- 저장 실패 시에는 실패를 표시하고 이전 상태로 유지하거나 재동기화한다.

## 8. 데이터 정합성 문제는 코드와 데이터 둘 다 해결한다

- 코드만 고치고 기존 데이터를 방치하지 않는다.
- 구조를 바꿨으면 기존 데이터 백필 필요 여부를 반드시 판단한다.
- “앞으로만 안 깨지게”로 끝내면 운영 중 같은 문제가 계속 보인다.

필수 질문:

- 기존 데이터 중 이미 깨진 건 없는가
- 있다면 저장 경로를 한 번 타면 회복되는가
- 아니면 별도 백필 스크립트가 필요한가

## 9. 권한 정책이 애매하면 더 좁게 연다

- 관리자만 바꿔야 하는 값인지, 회사도 바꿀 수 있는 값인지 먼저 정한다.
- 정책이 확정되지 않았는데 쓰기 경로를 여러 역할에 열어두지 않는다.
- 회사도 수정 가능해야 하면, 관리자와 같은 공통 쓰기 경로를 반드시 사용한다.

## 10. 성능 최적화는 원본 기준을 깨지 않는 선에서 한다

- 성능 때문에 single source of truth를 깨지 않는다.
- 조회 최적화가 필요하면 인덱스, 쿼리, 캐시를 사용한다.
- Firestore에서는 가능한 경우 `array-contains` 같은 원본 기반 쿼리를 먼저 검토한다.
- 성능용 캐시는 허용하지만, 진실을 대체하면 안 된다.

## 11. 마이그레이션 중에는 전환 상태를 명시한다

- 구조 변경이 들어가면 “현재 전환 중인지, 완료됐는지”를 문서로 남긴다.
- 임시 호환 코드가 있으면 이유, 제거 조건, 제거 위치를 기록한다.
- 전환 중 상태를 영구 상태처럼 두지 않는다.

## 12. 문서화 없는 구조 변경은 금지한다

- 비즈니스 원본이 바뀌는 변경은 문서가 같이 가야 한다.
- 최소한 아래는 남긴다.

필수 문서 항목:

- 원본 필드
- 파생 필드
- 읽기 기준
- 쓰기 진입점
- 백필 필요 여부
- 임시 호환 코드 존재 여부

## 13. 새 기능 추가 전 체크리스트

코드 작성 전 아래를 먼저 확인한다.

1. 이 상태의 원본은 어디인가
2. 같은 의미의 데이터를 다른 곳에도 저장하는가
3. 저장은 공통 함수 한 곳으로 들어가는가
4. 다른 화면도 같은 기준으로 읽는가
5. fallback가 원본을 우회하고 있지 않은가
6. 실패 시 에러가 드러나는가
7. 기존 데이터 마이그레이션이 필요한가

## 14. 리뷰 기준

리뷰할 때 아래 항목이 보이면 반드시 멈추고 다시 설계한다.

- 같은 의미의 데이터를 두 군데에서 각각 직접 수정함
- 데이터가 없을 때 다른 필드로 조용히 우회함
- 에러를 삼키고 빈 값으로 계속 진행함
- 화면마다 서로 다른 기준으로 같은 상태를 계산함
- 공통 유틸 없이 컴포넌트에서 문서를 직접 맞춤
- 임시 fallback가 영구 코드처럼 남아 있음

## 15. 이 프로젝트에서 지금 바로 적용할 원칙

- 회사 참여사업 원본: `companies.programs`
- 사업별 회사 목록: 원본에서 계산하거나 원본 기반 쿼리 사용
- `programs.companyIds`: 파생 인덱스
- 참여사업 변경: 공통 membership 유틸만 사용
- 데이터가 어긋났을 때: 숨기지 말고 수정 또는 백필

## 16. 한 줄 원칙

- 원본은 하나만 둔다.
- fallback로 깨진 상태를 가리지 않는다.
- 에러는 드러내고, 쓰기는 중앙화한다.
- 파생 데이터는 진실이 아니다.
